# Aleph MVP 产品架构与工作计划

## 摘要
Aleph 是一个独立产品，而不是某个底层 runtime 的附属功能。它的核心体验不是状态字段数量，而是一个更高阶的交互命题：

- 现实始终连续向前。
- 多个 `client/session` 代表不同人格，它们共享同一现实，但不共享同一主观世界。
- 每个人格拥有私有记忆、私有视角、私有判断方式。
- 人格切换不会重置现实；新的接管者必须面对前序人格留下的未完成事项、关系变化、风险、代价与机会。
- 系统允许有限共享，但共享服务于“共用同一现实”的产品隐喻，而不是退化成普通的多 agent 协作面板。

本 MVP 采用聊天优先、单前台人格、半自动切换、本地持久化。

## 核心设计原则
- 不把“疼痛、受伤、做饭”之类例子写死为底层 schema。
- 底层只抽象“连续现实 + 可继承后果”机制。
- 私有 memory 与 shared memory 是产品性机制，而不是通用技术装饰。
- 切换必须具备叙事意义和现实代价，不能只是 agent router。
- 优先保证人格差异、现实延续和接管解释性，而不是先追求技术复杂度。

## 三层模型
### Reality Layer
- 维护唯一连续的 `RealityThread`
- 记录当前 scene、open loops、recent events、active consequences
- 作为“现实真值”的主层，不承载人格私密解释

### Persona Layer
- `PersonaProfile` 定义人格身份、风格、专长、边界和可读写记忆域
- `PersonaSession` 表示一次实际接管实例
- `private memory` 保存该人格如何理解世界和自身角色
- `shared memory` 只保存共用现实所需的最低必要共识

### Orchestration Layer
- `ForegroundControl` 保证同一时刻只有一个前台人格
- `SwitchDaemon` 管理手动切换与半自动切换
- 每次切换都会留下解释和交接摘要

## 核心对象
### RealityThread
- 一条持续向前的现实主线
- 不是聊天记录，而是现实仍在推进的事实载体

### Consequence
- 尚未消化的现实影响项
- 不预设固定种类
- 可以表示未完成事务、关系变化、风险、资源占用、情境残留等

### SharedMemoryDomain
- 有限共享区，不是公共大脑
- 承载共同承认的事实、对外承诺、关系状态和共同约束

## MVP 交互形态
- 聊天优先
- 同一时刻只有一个前台人格与用户交互
- 系统需要可见：
  - 当前接管人格
  - 为什么发生切换
  - 当前现实里有哪些仍在延续的 consequences
  - 新人格接管时最需要知道什么

## 实现摘要
当前仓库中的 MVP 原型包含：

- SQLite 真值存储
- JSONL 调试日志
- 可运行的 `AlephEngine`
- 可运行的 `SwitchDaemon`
- 三个内置人格
- REPL 与 scripted scenario
- 针对连续现实、私有性、有限共享、可解释切换和单前台一致性的测试

## 默认假设
- 采用聊天优先的原型而不是多模态设备接入
- 采用单前台人格，不做多前台竞争
- 采用规则驱动的半自动切换，不做学习型策略
- shared memory 是“共用现实的最低必要共识层”，不是公共知识库
