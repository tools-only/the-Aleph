# Aleph v1 决策版计划

## 摘要

Aleph 是一个面向 AI agent 的类 infra 编排框架。它的核心价值不是“自身轻量”，
而是通过外层编译、编排与加速机制，让每个 `agent client runtime` 更快响应、
更容易隔离、更容易切换，并且不要求框架使用者手工拼接 prompt、memory 和 tools。

v1 固定约束：

- `one client = one real AI agent`
- client 是独立主体，而不是 subagent 模拟
- handoff 是核心机制
- 当前不引入 world model
- 为未来环境接入保留扩展口

## 核心架构

### 三层结构

- `Edge Gateway`
  采集交互事件、做轻量缓冲与流渲染。
- `Aleph Cloud`
  负责 client 编译、memory 投影、tool 投影、handoff、缓存、预热、观测。
- `Agent Runtime Pool`
  具体 agent runtime 通过 adapter 接入，例如 nanobot。

### 核心对象

- `ClientBlueprint`
- `ClientInstance`
- `AgentAdapter`
- `ProjectionCompiler`
- `SessionOrchestrator`
- `HandoffEnvelope`
- `Presentation Stream Event`

## Memory 与隔离

Aleph 在框架层做边界，而不是把隔离责任完全留给底层 agent。

- `Client-private memory`
  每个 client 独占。
- `Shared memory`
  跨 client 共识层，但必须受域和写入策略治理。
- `Handoff memory`
  专供接管过程使用。
- `Runtime metadata`
  trace、日志、cache 与评估。
- `Agent-native state`
  底层 agent 原生状态，受 adapter 管理并与 Aleph 明确同步边界。

## Runtime acceleration

Aleph 的复杂度主要用于加速 client runtime，而不是堆概念。

v1 必做：

- `Projection Cache`
- `Memory Slice Cache`
- `Prompt Skeleton Reuse`
- `Asynchronous Memory Maintenance`
- `Handoff Delta Packaging`
- `Candidate Client Prewarm`

硬规则：

- 所有预处理都必须无副作用
- 必须可取消
- 必须受 session 级预算约束

## 流式协议

内部编排事件和对外展示事件分离。

对外最小流式协议：

- `status`
- `delta`
- `tool_event`
- `handoff`
- `final`

## MVP 验收

- 至少支持一种真实 agent runtime，并可通过 adapter 扩展到第二种 runtime
- private memory 严格隔离
- shared memory 不会退化为公共上下文池
- handoff 只传递最小必要接管信息
- 缓存命中时首轮编排明显更快
- 长 session 不因上下文线性增长而显著拖慢
- 流式链路稳定输出五类事件
